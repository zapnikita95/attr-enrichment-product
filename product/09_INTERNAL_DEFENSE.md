## Ask к руководству

1. **Утвердить pricing:** batch = 1×MRR, recurring = 5–10%.
2. **Выделить analyst** на Studio-диагностику (0.25–0.5 FTE на активных партнёров).
3. **Разрешить пилот 0.2×MRR** для new partners (опционально).
4. **Не включать в базовый MRR** — отдельная строка в КП.

## FAQ для руководства

### Почему не включить в базовый MRR?

Это compute-heavy опция (GPU, LLM, аналитика). 1 MRR за batch покрывает затраты и даёт понятный upsell без размывания маржи базового поиска.

### Кто делает?

| Роль | Задачи |
|------|--------|
| Analyst | Studio, метрики, QBR |
| Eng | Extraction, upload, recurring |
| Sales + PM | Квалификация, КП, scope |

### Какой margin?

On-prem модели — нет API-токенов OpenAI. Основные затраты: GPU-время, analyst hours, поддержка recurring.

### Конкуренты?

- Ручная разметка контент-команды партнёра (медленно, дорого).
- SaaS PIM enrichment (без поисковой оценки, утечка данных).
- «Сделаем сами на ChatGPT» (не масштабируется, нет Dashboard path).

### Почему три инструмента, а не один UI?

Исторически три зрелых репо. Product hub (этот пакет) даёт единый narrative без рефакторинга ядра на этапе 1.

## Риски (честно)

| Риск | Митигация |
|------|-----------|
| Прогноз Studio ≠ post-launch CTR | Пилот 5–10k SKU, NDCG до/после |
| Vision quality зависит от фото | Phase1 prefilter, QA выборка |
| TOTP / доступ к Dashboard | UI заливки (порт 8766), инструкция партнёру |
| Малые delta ndcg на слайде | Пара Class I / zero / story |
| Шумные current_attrs | Audit first (Petrovich кейс) |
| GPU bottleneck | ollama-queue-proxy, планирование batch |

## Аргументы «за»

1. **Retention:** enterprise не уйдёт к конкуренту с «поиском + обогащением».
2. **Дифференциация:** замкнутый цикл оценка → заливка → метрики.
3. **Доказательства:** 7 кейсов, реальные цифры.
4. **Понятный чек:** 1×MRR без кастомных калькуляторов на каждую сделку.
