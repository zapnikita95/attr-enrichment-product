# Как выбираем тип фильтра (boolean / enum / multi_enum)

Это **не эвристика «мне так показалось»**. Цепочка вопросов одинакова для любого партнёра.
Цифры спроса (сколько ищут) — отдельно, из ClickHouse. Тип значения — из логики атрибута.

## Дерево решений

```
Q0. Атрибут вообще кандидат в ФИЛЬТР?
    — есть спрос в поиске (CH) ИЛИ стандартный facet fashion-каталога
    — НЕ дублирует уже заполненный param фида (collision)
    — значение можно стабильно нормализовать
    если нет → search-only / reject (не filter)

Q1. Атрибут отвечает на вопрос «есть / нет» про дискретную деталь или свойство?
    примеры: капюшон, карманы, опушка, подкладка, шнуровка как факт наличия
    → value_type = boolean
    → allowed_values = ["да", "нет"]  (один канон, без «есть/присутствует»)

Q2. Иначе: у атрибута есть небольшой набор взаимоисключающих вариантов,
    принятых в fashion / у партнёра в UI?
    примеры длины: mini | midi | maxi
    рукав: короткий | длинный | 3/4 | без рукавов
    застёжка: молния | пуговицы | кнопки | завязки | нет
    → value_type = enum
    → allowed_values = закрытый список (обычно 3–8, редко до 12)
    → один товар = одно значение

Q3. Иначе: на одном товаре могут одновременно быть несколько значений из набора?
    пример: принт «полоска + люрекс», «клетка + меланж»
    → value_type = multi_enum
    → тот же closed-set, multi=true

Q4. Иначе: число / непрерывная величина?
    → numeric_bins (интервалы), не сырые числа в facet
```

## Почему капюшон = boolean (обоснование)

1. **Онтология:** капюшон — деталь изделия, которая либо есть, либо нет.  
   Это не «вид капюшона» (меховой / съёмный) — такие подвиды, если понадобятся, = отдельный enum.
2. **Q1 = да** → boolean по правилу выше.
3. **Спрос (пруф, не причина типа):** в CH Zolla есть запросы вида «куртка с капюшоном», «худи» —  
   см. `FILTER_DEMAND_EVIDENCE.md`. Тип boolean выбран не потому что «так привычнее»,  
   а потому что вопрос покупателя бинарный: *нужен капюшон или нет*.
4. **Канон значений:** только `да|нет`. «С капюшоном» / «капюшон есть» — синонимы → coerce в `да`.

## Почему длина / рукав / застёжка = enum

- В fashion есть **стандартные варианты**, которые уже живут в фильтрах магазинов.
- Варианты **взаимоисключающие** на одном SKU (платье не mini и maxi одновременно).
- Список конечный → facet не взрывается.

## Почему принт = multi_enum

- На ткани может быть несколько признаков сразу (полоска + люрекс).
- Закрытый словарь узоров ограничивает кардинальность для UI.

## Что не делаем

- Не назначаем тип «потому что LLM так сказал» без прохождения Q1–Q4.
- Не делаем boolean из атрибута с 5+ осмысленными вариантами.
- Не делаем free-text facet («мягкий к телу»).
- Не дублируем цвет/размер/пол, если fill в фиде высокий (collision).
